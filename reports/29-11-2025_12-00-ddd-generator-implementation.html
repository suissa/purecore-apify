🏛️ Arquitetura DDD Implementada

O gerador cria uma estrutura completa seguindo os princípios do Domain-Driven Design:

Domain Layer

🏛️ Camada de Domínio

  • Entities: Entidades de domínio puras
  • Value Objects: Objetos de valor imutáveis
  • Aggregates: Agregados com raiz
  • Domain Events: Eventos de domínio
  • Domain Services: Regras de negócio
  • Repositories: Interfaces de persistência
Application Layer

📱 Camada de Aplicação

  • Application Services: Coordenação de operações
  • Commands: Operações de escrita (CQRS)
  • Queries: Operações de leitura (CQRS)
  • Command Handlers: Processamento de comandos
  • Query Handlers: Processamento de queries
  • DTOs: Data Transfer Objects
Infrastructure Layer

🔧 Camada de Infraestrutura

  • Repository Impl: Implementações concretas
  • Database Schema: Esquemas de banco
  • Database Context: Contextos de conexão
  • External Services: APIs externas
  • Configuration: Configurações técnicas
Presentation Layer

🌐 Camada de Apresentação

  • Controllers: Controladores REST
  • Routes: Definição de rotas
  • Middlewares: Middlewares específicos
  • Views: Templates/opcionais

🔄 CQRS Implementation

O gerador implementa Command Query Responsibility Segregation (CQRS) completo:

Commands: CreateUserCommand, UpdateUserCommand, DeleteUserCommand
Operações de escrita que modificam o estado
Queries: GetUserByIdQuery, GetAllUsersQuery, SearchUsersQuery
Operações de leitura que não modificam estado
Command Handlers: Processam comandos através do domínio
Coordenam validações e persistência
Query Handlers: Executam queries de forma otimizada
Podem usar caches, índices específicos, etc.

🚀 Benefícios da Arquitetura DDD

Separação de Responsabilidades: Cada camada tem responsabilidades bem definidas
Domain: regras de negócio | Application: coordenação | Infra: detalhes técnicos | Presentation: interface
Testabilidade: Domain layer completamente isolada e testável
Entidades, serviços de domínio e eventos podem ser testados independentemente
Manutenibilidade: Mudanças no domínio não afetam infraestrutura
Dependency Inversion Principle garante isolamento das camadas
Escalabilidade: Bounded Contexts permitem evolução independente
Cada contexto pode ser escalado e mantido separadamente
Ubiquitous Language: Linguagem unificada entre domain experts e developers
Código reflete diretamente os conceitos do domínio

🛠️ Como Usar o DDD Generator

1. Instalação e Configuração

import { autoGenerateDDDFromZodSchemas } from './src/auto-generator-ddd';

// Configurar geração DDD
await autoGenerateDDDFromZodSchemas({
modulesPath: 'src/modules',
domainPath: 'src/domain',
boundedContext: 'ecommerce',

verbose: true,
force: false,

dryRun: false

});

2. Schema Zod de Exemplo

// user.ts
import { z } from 'zod'; export const userSchema = z.object({

email: z.string().email(),
role: z.enum(['admin', 'user', 'moderator']),
createdAt: z.date(),
updatedAt: z.date()
});
export type User = z.infer;
